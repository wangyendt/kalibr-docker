# Kalibr 标定提速方案

## 背景

cam-cam 需要兼顾速度和可靠性。最初用 `--no-multithreading` 可以规避 rosbag 多进程读取异常，但会牺牲角点提取速度。实际复测发现，即使把 `cam.bag` 放到容器内部 `/tmp`，原始多进程路径仍然可能抛 `ROSBagFormatException`，因此根因不是共享盘 I/O，而是 worker 继承了父进程已打开的 rosbag 文件句柄。

## 已落地优化

`BagImageDatasetReader` 新增 `reopen()`，`TargetExtractor.multicoreExtractionWrapper()` 在 worker 启动时调用。这样每个 worker 都有独立 rosbag 句柄，避免多个进程共享同一个底层 open file description 并发 seek/read。

验证命令使用 `cam2cam_clahe`，强制 `--fast-extraction always` 后没有再出现 `ROSBagException` / `ROSBagFormatException`，角点提取为 `20/20`，报告显示 `Fallback used: False`。

wrapper 仍保留 `--fast-extraction auto`，作为生产默认安全网：先快跑；如果日志仍检测到 rosbag 多进程读取异常，自动归档 fast 结果并单线程重跑。

## cam-cam 提速建议

优先使用默认 `--fast-extraction auto`。在当前修复后，它通常等价于多进程快跑；只有异常时才回退。

数据量较大且环境稳定时可用 `--fast-extraction always` 做最快模式，但 CI/生产默认不建议跳过安全网。

尽量减少无效图片。Kalibr 优化阶段会对提取成功的视图建模，明显重复、模糊、覆盖区域高度相似的图片只会增加时间，质量收益有限。

可以考虑后续增加“视图去冗余采样”：先提角点并评估 3x3 区域、尺度、roll/pitch/yaw proxy，然后从候选图中选覆盖最均衡的一组进入 Kalibr。这会改变输入集合，需要作为显式参数，不应默认开启。

## cam-imu 提速建议

低风险提速项：

- 默认不要导出 poses，ProductionCalibration `private/zc-shuangzi-debug` 也没有导出 poses；Docker 命令用 `--no-export-poses`。
- 保持 `--dont-show-report`，只生成 PDF，不弹 GUI。
- 对外报告优先解析 `results-imucam.txt`，避免在质量检查阶段重复读取大文件。

中风险提速项：

- 如果时间同步已可靠，先用默认 `--no-time-calibration` 固定时间偏移，速度更快、优化变量更少；需要匹配旧 Production 输出时再加 `--estimate-time-offset --timeoffset-padding 0.04`。
- 增加 wrapper 参数透传 `--max-iter`，用于快速预检，例如先跑 10-15 次迭代看残差趋势，再用 30 次迭代跑正式结果。该项会改变收敛程度，不能默认降低。

高风险提速项：

- 降低 cam-imu 的 `poseKnotsPerSecond=100` 或 `biasKnotsPerSecond=50`。这会减少样条变量、明显加快 build/optimization，但会改变运动模型自由度，可能影响外参和时间偏移。需要做网格实验后才能作为参数开放。
- 对 IMU 做降采样或裁剪时间段。它会减少 residual 数量，但也会改变激励信息，必须在报告中标注，并和原始全量结果对比。

## 推荐路线

第一阶段已经完成 cam-cam 多进程句柄修复和 fallback 安全网。

第二阶段建议增加 cam-imu 的显式速度参数：`--max-iter`、`--pose-knots-per-second`、`--bias-knots-per-second`、`--imu-downsample-rate`。默认保持 Production 兼容值，只在用户明确指定时启用。

第三阶段做 cam-imu 网格实验：固定 `cam_imu_2` fixture 1，比较不同迭代次数、样条 knot rate、时间偏移开关和 IMU 降采样对耗时、T_ci、timeshift、gyro/accel residual 的影响，再决定哪些参数可以推荐。
